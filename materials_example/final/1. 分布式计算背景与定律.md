## 分布式计算的原因

### 数据规模爆炸与计算量估算

工业界的数据规模与课堂上完全不是一个量级：

| 环境 | 数据量 | 说明 |
|------|-------|------|
| 课堂学习 | MB 级 | Excel 文件，几千行 |
| 小公司 | GB 级 | 用户行为日志 |
| 工业界常见 | TB 级 | 电商平台日志 |
| 顶级互联网公司 | PB 级 | 搜索引擎索引 |

**具体案例：** 设数据集 $X \in \mathbb{R}^{10^9 \times 10^4}$（10亿行、1万特征），跑普通 OLS：

- $X$ 本身大小：$10^9 \times 10^4 \times 8\text{ bytes} \approx 80\text{ TB}$，根本无法载入内存
- 计算 $X^TX$ 的复杂度：$O(np^2) = O(10^{17})$ 次浮点运算
- 单机 10 GFLOPS 的 CPU 需要 $10^{17} / 10^{10} = 10^7$ 秒 ≈ **116 天**

> 理论性质再优雅，如果工程上算不出来，在现实层面几乎就是无用的。

### 存储墙（Memory Wall）

数据量远超单机内存容量，且**数据增速远快于内存增速**。这不是"买更大内存条"能解决的问题——工业界数据是持续产生、持续增长的。单机内存面临硬性天花板。

### Moore 定律放缓

过去"等几年硬件自然升级就能跑得动"的策略已经失效。单核 CPU 主频提升接近物理极限，现代芯片的策略是"做更多的核"而非"让一个核更快"。**靠等硬件升级解决计算问题，这条路越来越走不通。**

### 单机扩展（Scale-Up）的瓶颈

面对单机极限，第一直觉是"买更好的机器"（纵向扩展/Scale-Up）。极致的单机方案是 UMA 架构（见下）。

Scale-Up 有两个根本性瓶颈：

**痛点一：内存带宽瓶颈。** 所有 CPU 核心共享同一条数据总线，核心数越多、总线越拥堵，CPU 大量时间用于"空转等数据"而非计算。

**痛点二：极不经济。** 拥有 2TB 内存、128 核的大型服务器，价格可能超过 10 台普通服务器的总价，且性价比随摩尔定律放缓而持续下降。

### UMA 架构（统一内存访问）

UMA（Uniform Memory Access）是 Scale-Up 的代表架构：多个 CPU 核心共享同一块物理内存。

```
┌────────────────────────────────────┐
│         系统总线（共享）            │
└──┬────────┬────────┬──────────┬───┘
   │        │        │          │
 CPU1     CPU2     CPU3       共享内存
```

**优点：** 编程极简，所有核心直接读写同一块内存，多线程编程与单线程差异不大。

**本质限制：** 规模受限于单机总线带宽，只能扩展到几十到百核，无法无限横向扩展。

### 多机扩展（Scale-Out）与网络延迟

Scale-Out 即购买大量普通机器并通过网络相连，每个节点有**独立的私有内存**。这是分布式计算的核心思路。

**代价：** 节点间通信从主板总线变为网络，延迟大幅上升：

| 通信方式 | 典型延迟 |
|---------|---------|
| 读本地内存（L1缓存） | ~1 纳秒 |
| 读本地内存（RAM） | ~100 纳秒 |
| 同机房千兆以太网 | ~1 毫秒 |
| 跨数据中心网络 | ~10-100 毫秒 |

网络传输比读内存慢约 **100 万倍**。这就是为什么**在分布式时代，减少节点间通信的次数和数据量，往往比减少计算量更重要。** 有时宁愿让每台机器多算一些冗余，也不愿多发一次网络请求。

### Scale-Up vs Scale-Out 对比

| 对比维度 | Scale-Up（纵向扩展） | Scale-Out（横向扩展/分布式） |
|---------|-------------------|--------------------------|
| 内存模式 | 共享内存 | 私有内存，网络通信 |
| 扩展上限 | 低（受限于单机总线） | 极高（成千上万节点） |
| 通信延迟 | 纳秒级 | 毫秒级（慢约 10 万倍） |
| 容错性 | 差（单机故障即全崩） | 强（框架可自动重启任务） |
| 成本效益 | 高端机器性价比差 | 普通机器堆叠，成本可控 |

## 分布式计算的定义

**分布式计算：** 将一个庞大的计算任务拆解成若干子任务，分配给通过网络连接的多台计算机（节点）同时处理，协同完成计算目标。

类比：从"一个人做所有事"变成"一个团队分工协作"。但团队协作并不简单：任务能否拆分、沟通成本多高、是否存在必须等前一步完成才能开始的依赖关系——这些都是分布式系统的核心挑战。

### 并行计算 vs 分布式计算

| 对比维度 | 并行计算（多线程 / OpenMP） | 分布式计算（Spark / Ray） |
|---------|--------------------------|------------------------|
| 内存模式 | 共享内存 | 私有内存，网络通信 |
| 扩展规模 | 几十到百核 | 成千上万节点 |
| 通信延迟 | 纳秒级 | 毫秒级 |
| 容错性 | 极差（一线程崩即全崩） | 强（框架自动容错） |
| 开发挑战 | 死锁、竞态条件 | 序列化开销、网络拥塞、落后者问题 |
| 形象类比 | 同一间办公室共用白板 | 不同城市的办公室只能打电话 |

**优化目标的根本转变：**
- 单机时代：减少计算量（FLOP 数）
- 分布式时代：减少节点间通信次数和数据量

## 分布式计算定律

### Amdahl 定律：S(N) = 1 / [(1-P) + P/N]

**直觉：** 无论有多少台机器，程序里必须串行执行的部分始终需要那么长时间，它决定了加速的硬上限。

**符号含义：**
- $S(N)$：加速比，用 $N$ 台机器后程序快了多少倍
- $P$：程序中可完全并行化的代码比例
- $1-P$：必须串行执行的比例（如初始化、汇总结果）
- $N$：处理器/节点数量

**公式推导：** 设单机总耗时归一化为 1，串行部分耗时 $1-P$，并行部分耗时 $P$。用 $N$ 台机器后，串行部分不变，并行部分变为 $P/N$，总耗时为 $(1-P)+P/N$，故：

$$S(N) = \frac{1}{(1-P) + \dfrac{P}{N}}$$

**关键数值：**

| 并行比例 P | $N \to \infty$ 时的加速上限 |
|----------|--------------------------|
| 50% | 2 倍 |
| 90% | 10 倍 |
| 95% | 20 倍 |
| 99% | 100 倍 |

**四部分程序示例（课件原题）：**

| 部分 | 占比 | 并行度 $N_i$ |
|-----|------|------------|
| 部分1 | 11% | 1（完全串行） |
| 部分2 | 18% | 5 |
| 部分3 | 23% | 20 |
| 部分4 | 48% | 1.6 |

$$S = \frac{1}{0.11/1 + 0.18/5 + 0.23/20 + 0.48/1.6} = \frac{1}{0.11+0.036+0.0115+0.3} \approx 2.19$$

尽管大部分可以并行，整体只快了 2.19 倍，因为并行度低的部分拖了后腿。

**最重要启示：**
1. **程序的串行部分决定加速上限**，堆机器的收益快速递减
2. 算法设计核心目标：尽可能减少必须串行的部分，减少节点间同步等待（等待本质上也是串行）

### Gustafson 定律：S(N) = (1-P) + N·P

Amdahl 定律假设**问题规模固定**，用更多机器只是为了"做同样的事更快"。

Gustafson 定律换了一个视角：**有了更多算力，我们会去解决更大的问题。**

$$S(N) = (1-P) + N \cdot P$$

这个公式是**线性的**——加速比随 $N$ 线性增长，理论上没有天花板。

**DeepSeek 的例子：** 训练 671B 参数的模型需要至少几千张显卡。并非"一张卡不够快"，而是**野心也随算力增大**——有了更多显卡，就去训练更大的模型，参数量从 70B 增到 671B，$N \cdot T_{par}$ 这一项主导了全局，串行部分相对微不足道。

### Amdahl vs Gustafson 适用场景对比

| | Amdahl 定律 | Gustafson 定律 |
|--|------------|--------------|
| 假设 | 问题规模固定 | 问题规模随算力增大 |
| 问题 | 同样的工作，用更多机器做多快？ | 有了更多机器，能做多大的工作？ |
| 加速比 | 有硬上限 $\frac{1}{1-P}$ | 线性增长，无天花板 |
| 典型场景 | 批量推理、固定数据集训练 | 大模型预训练（LLM） |

## 分布式框架演进

### MPI（消息传递接口）

**年代：** 1990 年代，HPC（高性能科学计算）领域主流。

**工作方式：** 极其底层，程序员需**手动指挥每一组数据的发送和接收**，明确写出"节点 0 把数组前 1000 个元素发给节点 1"。

**优点：** 性能榨干到极致，无任何框架开销。

**致命缺点：无容错。** 1000 台机器跑三天，其中一台硬盘坏了，整个任务全军覆没，必须从头开始。在工业环境里，机器故障是常态，这不可接受。

**现状：** 仍在超算中心用于气象模拟、分子动力学等科学计算，互联网公司几乎不用。

### Hadoop MapReduce

**年代：** 约 2006 年，基于 Google GFS + MapReduce 两篇论文的开源实现。

**核心贡献：**

1. **HDFS（分布式文件系统）：** 把多台机器的硬盘虚拟成一个超大硬盘，数据自动存 3 份副本，机器故障时自动用副本恢复，用户无感。

2. **MapReduce 计算框架：** 把计算强制分成两阶段：
   - **Map：** 拆分数据，各节点并行处理自己的分片
   - **Reduce：** 汇总所有节点的局部结果
   
   自动容错：节点出错时，系统在其他节点自动重启该任务。

**致命缺点：磁盘 I/O 慢。** 每次 Map 和 Reduce 之间，中间结果必须**写回磁盘**。机器学习需要迭代几百次，每次都要"读磁盘 → 算 → 写磁盘"，磁盘比内存慢 100 倍以上，彻底拖垮速度。

**其他不足：** 表达能力较弱（部分操作难以仅用 Map 和 Reduce 完成）；不容易实现迭代计算；交互性不强。

### Spark（内存计算框架）

**年代：** 约 2010 年代，专门解决 Hadoop 磁盘 I/O 问题。

**核心创新：RDD（弹性分布式数据集）。** 中间计算结果**保存在内存**而非磁盘，对迭代型机器学习算法速度比 Hadoop 快 **10-100 倍**。

**致命缺点：BSP 同步架构导致落后者效应（Straggler Problem）。** Spark 采用整体同步并行（BSP）：每轮迭代必须**等所有节点都算完**才能进入下一轮。100 台机器里有 1 台因网络抖动变慢，其余 99 台只能空转等待，严重浪费资源。

### Ray（动态任务图框架）

**年代：** 2017 年，UC Berkeley RISELab 开发，专为现代 AI 工作负载设计。

**核心优势：**

1. **不强制编程模型：** 不像 MapReduce 强迫拆成两步，Ray 支持任意逻辑
2. **极低延迟：** 毫秒级任务调度，适合强化学习等实时场景
3. **Python 原生：** 普通 Python 函数加 `@ray.remote` 装饰器即可并行
4. **单机调试、多机部署：** 同一套代码无需修改即可从笔记本扩展到集群

```python
@ray.remote
def compute(x):
    return x ** 2

# 4 个任务同时在不同核心/机器上运行
results = ray.get([compute.remote(i) for i in range(4)])
```

### 四代框架对比表

| 特性 | MPI | Hadoop | Spark | Ray |
|------|-----|--------|-------|-----|
| 出现年代 | 1990s | ~2006 | ~2010 | 2017 |
| 核心存储 | — | 磁盘（HDFS） | 内存（RDD） | 内存（Object Store） |
| 编程模型 | 手动消息传递 | 严格 Map-Reduce | 算子 DAG | 通用异步 Task / Actor |
| 容错性 | 无 | 强（磁盘副本） | 强 | 强 |
| 适用场景 | HPC 科学计算 | 离线批处理 | 传统 ML、流处理 | 深度学习、RL、最优化 |
| 致命缺点 | 无容错 | 磁盘 I/O 慢 | 落后者效应（BSP） | — |
