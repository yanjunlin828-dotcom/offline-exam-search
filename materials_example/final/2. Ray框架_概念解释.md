## Ray 集群架构

Ray 集群由**一个头部节点 + 若干工作节点**组成。

```
           ┌─────────────────────────────┐
           │      头部节点（Head Node）   │
           │  调度器 / GCS / Driver 程序  │
           └────────────┬────────────────┘
                        │ 网络
          ┌─────────────┴──────────────┐
    ┌─────┴──────┐             ┌───────┴──────┐
    │  工作节点 1 │             │  工作节点 2  │
    │ Raylet     │             │ Raylet       │
    │ Object Store│            │ Object Store │
    └────────────┘             └──────────────┘
```

### Head Node（头节点）

头节点是集群的"大脑"，负责全局调度与状态管理：
- **Driver 程序**在此运行（你写的 Python 代码）
- **Scheduler（调度器）**：决定把哪个任务发给哪个工作节点
- **GCS（Global Control Store）**：记录全局状态（哪些节点在线、哪些对象在哪里）
- **Autoscaler**：根据负载自动增减工作节点数量
- **Dashboard**：`http://127.0.0.1:8265` 可视化集群状态

### Worker Node（工作节点）

工作节点是"打工人"，只负责执行具体计算：
- **Worker Process**：真正运行你的远程函数
- **Raylet**：本地调度器，接收来自头节点的任务分配
- **Object Store（Plasma）**：本节点的共享内存，存放中间结果

### GCS（全局控制存储，Global Control Store）

GCS 是整个集群的"中枢神经"，维护对象位置映射表（某个 ObjectRef 对应的数据在哪个节点的 Object Store 里）、节点健康状态等全局元数据。

### Object Store（分布式对象存储）

每个节点都有自己的本地 Object Store（基于 Plasma 共享内存）。`.remote()` 的返回值、`ray.put()` 上传的数据，都存在这里。Object Store 支持跨节点数据访问，但同节点内访问是**零拷贝**的。

### 零拷贝读取（Zero-Copy Read）

当多个任务在**同一台机器**上读取同一份数据时，它们直接读同一块内存地址，完全不需要复制数据。这避免了大矩阵的重复序列化开销。

> 零拷贝的触发条件：数据通过 `ray.put()` 存入本地 Object Store，同节点多任务同时读取。

## Ray 核心 API

### ray.init() / ray.shutdown()

`ray.init()` 启动 Ray 运行时（单机模式）或连接已有集群（集群模式）。调用后 Ray 才能接受任务提交。`ray.shutdown()` 释放所有资源。

```python
ray.init()              # 单机模式，自动使用所有可用 CPU
ray.init(num_cpus=4)    # 限制使用 4 个核
ray.init()              # 集群模式：自动检测并连接集群
```

**最重要的特性：** 同一套代码，在笔记本上单机调试和在真实集群上运行**不需要修改任何代码**——只需要 `ray.init()` 连接到不同的目标。

### @ray.remote 装饰器

`@ray.remote` 是把普通 Python 函数（或类）变成"可以远程调度"的核心手段。

- 加在函数上 → 变成 **Ray Task**（无状态远程函数）
- 加在类上 → 变成 **Ray Actor**（有状态远程对象）

装饰后的函数/类不再直接可调用，必须通过 `.remote()` 才能执行。

### .remote() 调用（非阻塞异步提交）

`.remote()` 是提交任务的方式。调用后：
1. 任务**立即被提交**到调度器队列
2. **立即返回**一个 ObjectRef（不等任务完成）
3. 任务在后台某个工作节点上异步执行

```python
ref = compute.remote(x)   # 瞬间返回，不等结果
# 主程序可以继续做其他事情...
```

**时间效益：** 10 个任务各需 5 秒，串行需 50 秒；全部 `.remote()` 后统一等待，只需约 5 秒（取决于最慢的那个）。原理：总时间从 $\sum t_i$ 变成 $\max t_i$。

### ray.get()（阻塞等待结果）

`ray.get()` 用 ObjectRef 取回真实计算结果。如果任务还没完成，主程序在此**阻塞等待**；如果已完成，立刻返回结果。

```python
result = ray.get(ref)         # 等单个任务
results = ray.get([r1, r2])   # 等多个任务（推荐批量调用）
```

`ray.get()` 是整个 Ray 编程模型中**唯一的同步点**，必须谨慎放置——放错位置会导致并行退化为串行（见"Ray 常见陷阱"）。

### ray.put()（手动上传到 Object Store）

`ray.put()` 把一个大对象提前存入 Object Store，返回 ObjectRef。后续任务只需传这个 ID，而不是传数据本身。

**解决的问题：** 如果直接把 800MB 的大矩阵作为参数传给 10 个任务，Ray 会复制 10 份、发送 8GB 数据。用 `ray.put()` 只存一份，10 个任务共享同一份数据，网络流量从 8GB 降到接近 0（同节点零拷贝）。

```python
data_ref = ray.put(big_matrix)          # 只存一份
refs = [task.remote(data_ref, i) for i in range(10)]  # 传 ID，不传数据
```

### ObjectRef（对象引用）

ObjectRef 是 `.remote()` 或 `ray.put()` 的返回值，本质上是一个**"取件码"**，不包含真实数据，只是指向 Object Store 中某个对象的引用。

**Future 模式：** ObjectRef 是"未来结果的承诺"——任务还在计算，但你已经拿到了"将来会有结果"的凭证，想取的时候用 `ray.get()` 兑换。ObjectRef 也可以直接作为参数传给下一个 `.remote()` 调用，Ray 会自动解析依赖。

## Ray Task（无状态远程函数）

### Ray Task 特性与适用场景

**无状态的含义：** 输出只由输入决定，和任何外部变量、历史调用次数无关。相同输入永远给出相同输出。

```python
# 无状态（可以用 Task）
@ray.remote
def square(x):
    return x * x   # 只依赖输入 x

# 有状态（不能用 Task！依赖外部的 counter）
counter = 0
def count():
    global counter
    counter += 1
    return counter   # 输出取决于"被调用过几次"
```

**为什么无状态对分布式很重要：** Ray 调度器不知道你的代码逻辑，它把任务发给"任何一台空闲的机器"。如果任务有状态（依赖某台特定机器上的全局变量），发错机器就出错。无状态任务可以被安全地发给任意节点。

**天然适合 Ray Task 的场景（简单并行 / Embarrassingly Parallel）：**

| 场景 | 为什么是简单并行 |
|------|----------------|
| 蒙特卡洛模拟 | 每次用不同随机种子，彼此独立 |
| Bootstrap 重抽样 | 每次有放回抽样互相独立 |
| K 折交叉验证 | 每个 fold 的训练互相独立 |
| 超参数网格搜索 | 每组参数的训练互相独立 |
| 随机森林 | 每棵树的训练互相独立 |

### 简单并行的实现陷阱

#### Bootstrap 重抽样：随机数种子管理

在分布式 Bootstrap 中，不同 Task 运行在不同机器上。若不显式设定种子，所有节点可能共用相同的默认随机种子，导致 $B$ 个自举数据集**完全相同**，结果毫无意义。

**正确做法：** 在每个 Task 内部，用任务编号作为种子：

```python
@ray.remote
def bootstrap_task(data_ref, seed):
    np.random.seed(seed)          # 每个 Task 独立种子，不可省略
    data = ray.get(data_ref)
    idx = np.random.choice(len(data), len(data), replace=True)
    return compute_statistic(data[idx])

data_ref = ray.put(dataset)
refs = [bootstrap_task.remote(data_ref, seed=i) for i in range(B)]
```

#### K 折交叉验证：用 ray.put() 避免重复传输数据

K 折 CV 中每个 Task 都需要完整数据集。若直接把数据作为参数传给 `.remote()`，Ray 会**复制 K 次**发送 K 份数据，瞬间打爆网络带宽。

**正确做法：** 提前 `ray.put()` 一次，所有 Task 共享同一份引用：

```python
data_ref = ray.put(dataset)    # 只传一次到 Object Store
refs = [cv_fold.remote(data_ref, fold_id=k) for k in range(K)]
```

## Ray Actor（有状态远程对象）

### Actor 定义（@ray.remote class）

Ray Actor 是把 Python **类**加上 `@ray.remote` 装饰器，变成一个**常驻远程进程中的有状态对象**。

```python
@ray.remote
class Counter:
    def __init__(self):
        self.count = 0      # 内部状态

    def increment(self):
        self.count += 1

    def get(self):
        return self.count
```

与 Task 不同，Actor 在一台特定工作节点上**持续存在**，所有方法调用都在同一个进程里执行，因此可以安全地维护内部状态。

### Actor Handle 与方法调用（.method.remote()）

创建 Actor 实例返回一个 **Actor Handle**（句柄），之后所有方法调用都通过这个 handle：

```python
c = Counter.remote()         # 创建 Actor，返回 handle（不是对象本身！）
c.increment.remote()         # 调用方法，异步提交
c.increment.remote()
val = ray.get(c.get.remote()) # 取回内部状态
# val == 2
```

Actor handle 可以传递给其他任务，允许多个任务并发访问同一个 Actor。

### Actor 内部状态持久化

Actor 的所有方法调用**串行执行**（在同一进程内按提交顺序排队）。这保证了状态更新的安全性，不会有竞态条件。状态在 Actor 的整个生命周期内持续保留，只要 handle 还存在，Actor 就不会被销毁。

### Actor vs Task 选择依据

| 场景 | 推荐 | 原因 |
|------|------|------|
| 输出只依赖输入 | Task | 无状态，任意调度，并行度最高 |
| 需要跨多次调用积累状态 | Actor | 只有 Actor 能在调用间保持状态 |
| 参数服务器（全局模型参数） | Actor | 所有 Worker 需要读写同一份参数 |
| ADMM Worker（保存 xᵢ, uᵢ） | Actor | 每个 Worker 在多轮迭代中保持自己的局部变量 |
| 分布式数据库 / 缓存 | Actor | 需要有状态的持久服务 |

### Actor 在分布式优化（ADMM / 参数服务器）中的角色

在一致性 ADMM 中，每个 Worker 需要在多轮迭代中保存自己的局部变量 $\theta_i$ 和对偶变量 $u_i$，这正是 Actor 的典型使用场景：

- 每个 **ADMM Worker = 一个 Actor**，持有 $\theta_i$、$u_i$、预分解的 Cholesky 因子
- 主进程（Driver）协调每轮的 z-update（计算均值、软阈值）
- Worker Actor 在每轮收到新的 $z^k$ 后独立更新 $\theta_i$，**完全并行**

在参数服务器架构中，**Parameter Server = 一个 Actor**，持有全局模型参数；各 Worker Task 拉取参数、计算梯度、推送更新。

## Ray 常见陷阱

### 过早调用 ray.get() 导致串行化

这是 Ray 编程中最常见、最致命的错误：

```python
# ❌ 错误：每次提交后立刻等待，完全串行
for i in range(10):
    ref = compute.remote(i)
    result = ray.get(ref)    # 卡在这里，等这一个任务完成才继续
    results.append(result)
# 总耗时 = 10 × 单任务时间（和不用 Ray 一样慢！）

# ✅ 正确：先全部提交，再统一等待
refs = [compute.remote(i) for i in range(10)]  # 全部异步提交
results = ray.get(refs)                         # 统一等待
# 总耗时 ≈ 单任务时间（真正并行）
```

**记忆口诀：先批量 `.remote()`，后统一 `ray.get()`。**

### Map-Reduce 反模式 vs 正确写法

正确的 Map-Reduce 三步：

```
Step 1 Map：   refs = [f.remote(x) for x in data]   ← 批量发出，不等待
Step 2 Barrier：results = ray.get(refs)              ← 统一等待所有任务
Step 3 Reduce： answer = aggregate(results)          ← 汇总（如 np.sum / np.mean）
```

**反模式**：在 Map 循环内部调用 `ray.get()`，任务变成一个接一个串行执行，并行效果完全丢失。

## Ray 集群部署

### 单机模式 vs 多机集群模式

| 模式 | 场景 | 启动方式 |
|------|------|---------|
| 单机模式 | 开发、调试 | `ray.init()` 或 `ray.init(num_cpus=N)` |
| 集群模式 | 生产、大规模 | 先启动集群，再 `ray.init()` 连接 |

**核心优势：** 同一套代码，无需修改，单机开发完毕直接放到集群运行。

### 集群配置与节点管理

```bash
# 在头节点机器上
ray start --head --port=6379

# 在每台工作节点机器上（填写头节点的 IP）
ray start --address="192.168.1.5:6379" --num-cpus=4

# 查看集群状态
ray status

# 停止节点
ray stop
```

也可以通过 `http://头节点IP:8265` 打开可视化 Dashboard，查看各节点资源使用情况和任务运行状态。
