> 对应 Lec 15。在带正则项一致性 ADMM 基础上延伸至更一般的优化框架（PPG），并涵盖现代分布式机器学习的三个前沿方向：异步优化、联邦学习、大模型分布式训练。前置：单机 ADMM 见"7. 非光滑优化与ADMM"，分布式 Lasso 见"8. 一致性优化"。

## PPG（Proximal-Proximal-Gradient）方法

### ADMM 无法处理的场景（目标函数中同时有多个非光滑项）

带正则项一致性 ADMM 能处理的问题结构是：

$$\min_x \sum_{i=1}^N f_i(x) + g(z), \quad \text{s.t. } x_i = z$$

**ADMM 遇到的困境——两个非光滑项同时出现：** 例如非负稀疏线性回归：

$$\min_\beta \frac{1}{2}\|A\beta - b\|_2^2 + \lambda\|\beta\|_1 + \underbrace{I_{\beta \geq 0}(\beta)}_{\text{非负约束（又一个非光滑项）}}$$

$\lambda\|\beta\|_1$ 和非负约束 $\beta \geq 0$ 都是非光滑的，ADMM 的变量分裂方案需要嵌套处理，无法干净地把两个非光滑项分开——ADMM 的 $z$ 更新子问题本身变得很难解。

**一般结构（PPG 的出发点）：**

$$\min_x \underbrace{r(x)}_{\text{全局非光滑}} + \sum_{i=1}^N \left[\underbrace{f_i(x)}_{\text{本地光滑}} + \underbrace{g_i(x)}_{\text{本地非光滑}}\right]$$

| 项 | 类型 | 实例 |
|---|---|---|
| $r(x)$ | **全局**非光滑（有近端算子）| $\lambda\|x\|_1$（Lasso）|
| $f_i(x)$ | **本地**光滑（可求梯度）| $\frac{1}{2}\|A_i x - b_i\|^2$ |
| $g_i(x)$ | **本地**非光滑（有近端算子）| $I_{x \geq 0}(x)$（非负约束）|

ADMM 只支持一个全局非光滑项；PPG 允许每个节点还有各自的 $g_i$。

### PPG 的三步迭代结构（近端梯度步 + 两个近端算子）

**步骤 1（主节点）：全局近端算子——处理 r(x)**

$$\bar{z}^k = \frac{1}{N}\sum_{i=1}^N z_i^k$$

$$\boxed{\beta^{k+1/2} = \text{prox}_{\alpha r}\!\left(\bar{z}^k\right)}$$

例如 $r = \lambda\|\cdot\|_1$ 时，$\beta^{k+1/2} = S_{\alpha\lambda}(\bar{z}^k)$（软阈值）。主节点广播 $\beta^{k+1/2}$ 给所有工作节点。

**步骤 2-4（各工作节点，完全并行）：梯度步 + 本地近端算子——处理 f_i 和 g_i**

工作节点 $i$ 依次执行：

$$\nabla f_i(\beta^{k+1/2}) = A_i^T(A_i\beta^{k+1/2} - b_i) \quad \text{（梯度步：利用本地光滑损失）}$$

$$v_i^k = 2\beta^{k+1/2} - z_i^k - \alpha\nabla f_i(\beta^{k+1/2}) \quad \text{（梯度修正 + 动量）}$$

$$\boxed{\beta_i^{k+1} = \text{prox}_{\alpha g_i}(v_i^k)} \quad \text{（本地近端算子：处理局部非光滑项）}$$

$$z_i^{k+1} = z_i^k + \beta_i^{k+1} - \beta^{k+1/2} \quad \text{（对偶变量更新）}$$

工作节点将 $z_i^{k+1}$ 发回主节点。

**通信量：** 每轮仅传递 $p$ 维向量，与 ADMM 一致。

### PPG vs ADMM 的适用范围对比

| 维度 | ADMM（含正则项版）| PPG |
|------|---|---|
| 目标函数 | $\sum f_i(\theta_i) + g(z)$ | $r(x) + \sum [f_i(x) + g_i(x)]$ |
| 非光滑项个数 | 通常 1 个全局 $g$ | 1 个全局 $r$ + 每节点各 1 个 $g_i$ |
| $x/\theta_i$ 子问题 | 解增广拉格朗日子问题（可能需要线性方程组）| 计算梯度 + 近端算子（各一次）|
| 框架基础 | 增广拉格朗日 + 变量分裂 | 近端梯度 + 分裂迭代 |
| 典型适用 | Lasso、岭回归、SVM | 非负稀疏回归、Elastic Net、Group Lasso |
| 非负约束 | 困难（两个非光滑项嵌套）| 自然处理（$\text{prox}_{g_i} = \max(\cdot, 0)$）|

## 异步优化（Async SGD）与延迟梯度

### 同步算法的瓶颈（Straggler Problem）

**同步 SGD/ADMM 的工作模式：**

```
所有 Worker 同时计算梯度/解子问题
         ↓（等待所有节点完成）
Head 聚合结果并更新参数
         ↓
进入下一轮（全员同步出发）
```

**掉队者（Straggler）导致的等待：** 每轮的总时间由**最慢节点**决定：

$$T_{\text{每轮}} = \max_i T_i$$

100 个节点中最慢的 1 个需要 10 倍时间，整个集群吞吐量降至 10%。速度差异来源：CPU/GPU 性能差异、数据块大小不均、网络拥塞、其他进程抢占资源。

**集群规模越大，遇到"最慢节点"的概率越高，掉队者问题越严重。**

### 异步 SGD 的工作方式（不等落后者，直接用过期梯度更新）

**异步算法的关键设计：** 工作节点算完就提交，主节点收到就更新，不等其他节点。

```
Worker A 算完 → 立刻发送 → Head 收到即更新参数
Worker B 还在算...
Worker A 已经开始下一轮计算（用最新参数）
Worker B 才算完 → 发送（此时 Head 参数已更新了若干步）
```

**主节点更新规则（异步 SGD）：**

$$w_{t+1} = w_t - \alpha \cdot g_{t-\tau}$$

其中 $g_{t-\tau}$ 是某个工作节点用 $\tau$ 步之前的旧参数 $w_{t-\tau}$ 计算出的梯度。

### 延迟梯度（Stale Gradient）问题（用旧参数算出的梯度误差）

**时间轴理解：**

```
全局参数时间轴：  w₀ → w₁ → w₂ → ... → wₜ → wₜ₊₁ → ... → wₜ₊τ

Worker(m) 的时间轴：
  时刻 t：     看到全局参数 wₜ，开始计算梯度 gₜ = ∇ℓ(wₜ)
  τ 步后：     梯度算好了，发送给主节点
  主节点收到时：全局已更新到 wₜ₊τ（其他节点已更新了 τ 步）
  → 主节点使用 gₜ 更新 wₜ₊τ（梯度延迟了 τ 步！）
```

**延迟梯度带来的问题：** $g_{t-\tau} = \nabla\ell(w_{t-\tau})$ 是在旧参数处计算的梯度，而真正应该用的是当前参数处的梯度 $\nabla\ell(w_t)$。延迟 $\tau$ 越大，方向偏差越大，更新的步子可能走错方向。

**延迟 $\tau$ 的大小取决于：** 节点数量 $N$（越多延迟越大）、节点间速度差异、网络延迟。

### 异步 SGD 的核心权衡（速度快但收敛不稳定）

| 维度 | 同步 SGD | 异步 SGD |
|------|---|---|
| 单位时间更新次数 | 少（每轮等最慢节点）| 多（算完即更新，无等待）|
| 梯度新鲜度 | 新鲜（总用最新参数）| 可能过时（有延迟 $\tau$）|
| 收敛理论 | 成熟（标准凸/非凸分析）| 复杂（需显式考虑延迟效应）|
| 调参难度 | 相对容易 | 学习率需要更小（补偿噪声）|
| 适合场景 | 节点速度均衡、通信快 | 节点异构、通信不稳定、大集群 |
| 工程实现 | 简单（全局 Barrier）| 复杂（参数服务器异步版本）|

**核心结论：** 异步算法用"梯度更新频率更高"换取"每次更新方向不够准确"。网络带宽有限、节点速度差异大时，异步优于同步；对收敛精度要求高时，同步更可靠。

## 联邦学习（Federated Learning）

### 联邦学习的动机（数据隐私保护、数据不离开本地）

**传统分布式学习的假设：** 数据可以集中到一个受控集群中（数据中心），节点由同一机构管理。

**联邦学习面对的新场景：**

| 特征 | 说明 |
|------|------|
| 数据分散 | 数据分布在大量设备（手机、医院、工厂）中 |
| 数据不可集中 | 原始数据不能上传到中心服务器（隐私法规 GDPR、商业机密）|
| 数据量不均衡 | 各客户端数据量差异可达百倍 |
| 数据分布异构 | 各客户端数据分布严重不同（医院的病人类型不同）|
| 设备异构 | 手机性能差异大，网络时好时坏，随时上线和掉线 |

**核心目标：数据不动，模型动。** 原始数据永远留在本地，只传输模型参数或梯度。

### FedAvg 算法（本地多步 SGD + 全局参数平均聚合）

**FedAvg（Federated Averaging）算法流程：**

**第 1 步（服务器）：** 将当前全局模型 $w^t$ 广播给随机选取的 $K$ 个客户端

**第 2 步（各被选中客户端，并行）：** 用本地数据训练 $E$ 个 epoch（多步 SGD），得到本地更新后的模型 $w_k^{t+1}$

**第 3 步（客户端）：** 将更新后的参数 $w_k^{t+1}$ 上传给服务器

**第 4 步（服务器）：** 按各客户端样本量 $n_k$ 加权平均：

$$\boxed{w^{t+1} = \sum_{k=1}^K \frac{n_k}{n} w_k^{t+1}}, \quad n = \sum_{k=1}^K n_k$$

**FedAvg 与分布式 GD 的本质区别：**

| 对比维度 | 分布式 GD | FedAvg |
|---------|---|---|
| 每轮本地步数 | 1 步梯度更新 | $E$ 个 epoch（可能几十步）|
| 通信频率 | 每步都同步 | 每 $E$ epoch 同步一次 |
| 通信减少 | 参考基准 | 减少约 $E$ 倍 |
| 精度影响 | 无（每步都一致）| 有（本地多步可能偏向局部分布）|

FedAvg 用更少的通信换取可接受的精度损失——在移动设备上尤为重要（上传流量昂贵）。

### Non-IID 数据分布的挑战（各节点数据分布不同导致模型漂移）

**i.i.d. 假设：** 传统机器学习假设所有训练数据来自同一分布（独立同分布）。

**联邦学习的现实：** 各客户端数据分布差异显著。

**具体例子：** 手机键盘预测模型
- 用户 A（程序员）：数据中大量 Python/Java 关键词
- 用户 B（文学爱好者）：数据中大量文学词汇
- 用户 C（商务人士）：数据中大量商务英语短语

本地模型各自优化时，会偏向自己的数据分布——这称为 **Client Drift（客户端漂移）**。

**Client Drift 的后果：** 加权平均时，各客户端的"更新方向"指向不同的局部最优，平均后的全局模型可能对所有人都不是最优的，甚至偏离全局最优。

**研究应对方向：**

| 方向 | 代表方法 | 思路 |
|------|------|------|
| 正则化拉回 | FedProx | 本地训练时加正则项 $\frac{\mu}{2}\|w - w^t\|^2$，防止本地模型跑太远 |
| 方差修正 | SCAFFOLD | 用控制变量修正本地梯度方向，对齐到全局梯度 |
| 个性化 | Per-FedAvg | 为每个用户学习个性化模型（全局模型只作为初始点）|

### 通信稀疏化（减少通信轮次与通信量）

**两个层面的通信优化：**

| 优化目标 | 策略 | 代价 |
|---------|------|------|
| 减少通信**轮次** | FedAvg 的本地多步训练 | Non-IID 导致精度下降 |
| 减少每轮通信**量** | 梯度/参数压缩：量化（用 8-bit 代替 32-bit）、稀疏化（只传最大梯度）| 压缩误差 |

**梯度压缩的权衡：** 减少上传量（对移动设备友好），但引入压缩误差，可能降低收敛精度或速度。

## 大模型分布式训练

### 数据并行（Data Parallelism）：每个节点持有完整模型副本

**工作方式：** 每张 GPU 持有**完整的模型副本**，不同 GPU 处理不同批次的数据。

```
GPU 0: 持有完整模型 → 前向+反向计算 → 得到梯度 g₀
GPU 1: 持有完整模型 → 前向+反向计算 → 得到梯度 g₁
...
GPU K: 持有完整模型 → 前向+反向计算 → 得到梯度 gₖ

梯度 All-Reduce: g = (g₀ + g₁ + ... + gₖ) / K
所有 GPU 用相同的 g 更新模型参数
```

**通信方式：** Ring All-Reduce（见"6. 光滑优化"）——通信量 $O(p)$，与节点数 $K$ 无关。

**限制：** 每张 GPU 必须能容纳**完整的模型参数 + 梯度 + 优化器状态**。对于 70B 参数的模型，光参数就需要约 140 GB（FP16），远超单卡容量。

### 模型并行（Model Parallelism）：模型层分布在不同节点上

**张量并行（Tensor Parallelism）：** 将单个矩阵乘法切分到多张 GPU。

```
Y = X · W   (X: m×k, W: k×n)
→ W 按列切分: W = [W₁ | W₂]，W₁在GPU 0，W₂在GPU 1
→ GPU 0 计算 Y₁ = X·W₁，GPU 1 计算 Y₂ = X·W₂
→ All-Reduce 拼接: Y = [Y₁ | Y₂]
```

**要求：** GPU 之间通信延迟极低（NVLink/InfiniBand），否则通信开销淹没计算收益。适合同一台服务器内的多 GPU。

### 流水线并行（Pipeline Parallelism）：微批次流水线

**基本思想：** 将模型的不同**层（Layer）**放到不同 GPU，数据像流水线一样流过。

```
GPU 0: 持有 Layer 1-8
GPU 1: 持有 Layer 9-16
GPU 2: 持有 Layer 17-24
GPU 3: 持有 Layer 25-32

Batch 1:  GPU0 算完 → 传给 GPU1 → ... → GPU3 算完
Batch 2:  GPU0 算完 → 传给 GPU1 → ...
（GPU0 算 Batch 2 时，GPU1 在算 Batch 1）
```

**流水线气泡（Pipeline Bubble）：** 流水线启动和排空阶段有 GPU 空闲——前 $K-1$ 步和后 $K-1$ 步只有部分 GPU 在工作（$K$ = GPU 数量）。气泡比例：$\frac{K-1}{M+K-1}$，$M$ 为微批次数量，$M$ 越大气泡越少。

**适用场景：** 跨机器的并行（GPU 卡之间通过网络相连，延迟较高）。

### 三种并行方式的对比与组合（3D 并行）

| 并行方式 | 切分对象 | 通信内容 | 适用场景 |
|---------|---|---|---|
| 数据并行 | 数据（每卡有完整模型）| 梯度 All-Reduce | 模型能放进单卡时 |
| 张量并行 | 单层矩阵（跨卡切分）| 中间激活值 | 同台机器内多 GPU |
| 流水线并行 | 层（跨卡分层）| 激活值（前向）+ 梯度（反向）| 跨机器多节点 |

**3D 并行（实际大模型训练采用）：** 三种并行方式同时使用。

```
典型配置（训练 GPT-3 175B 规模）：
  数据并行: 8 路（8 组，每组独立处理不同数据）
× 流水线并行: 8 路（8 台机器，各持有不同层）
× 张量并行: 8 路（每台机器内 8 张 GPU，切分矩阵乘法）
= 总计 512 张 GPU
```

## ZeRO 优化器（Zero Redundancy Optimizer）

### 为什么优化器状态也需要分片（Adam 状态占模型参数 3 倍内存）

**以 Adam 为例，训练时每个参数 $w$ 需要存储：**

| 存储内容 | 量 | 说明 |
|---------|---|------|
| 参数本身 $w$ | 1× | 模型权重 |
| 梯度 $g$ | 1× | 反向传播结果 |
| 一阶矩 $m$（动量）| 1× | Adam 状态 |
| 二阶矩 $v$（自适应学习率）| 1× | Adam 状态 |

**总计 4 倍参数量。** 用 FP32（32-bit）训练 70B 参数的模型：$70 \times 10^9 \times 4 \times 4\text{ bytes} = 1.12 \text{ TB}$，远超任何单卡显存。

**数据并行的冗余：** 8 路数据并行 = 8 张 GPU 各自存储一份完整的参数 + 梯度 + 优化器状态，其中 7/8 是完全重复的冗余！

**ZeRO 的思路：** $G$ 路数据并行时，每张 GPU 只保存 $1/G$ 的状态，需要时通过通信获取其他卡的部分。

### ZeRO Stage 1：分片优化器状态（OS）

**只切分优化器状态**（$m$ 和 $v$），参数和梯度仍然每卡完整持有。

```
GPU 0: 完整参数 + 完整梯度 + [优化器状态 第 0..p/G 部分]
GPU 1: 完整参数 + 完整梯度 + [优化器状态 第 p/G..2p/G 部分]
...
GPU G: 完整参数 + 完整梯度 + [优化器状态 第 (G-1)p/G..p 部分]
```

**内存节省：** 优化器状态占总内存的约 50%（$2/4$），切 $G$ 份后单卡优化器状态内存降至 $1/G$。

**通信额外代价：** 每轮参数更新后做一次 All-Gather，让各 GPU 拿到完整参数。

### ZeRO Stage 2：分片优化器状态 + 梯度

在 Stage 1 基础上，**也切分梯度**。

```
GPU 0: 完整参数 + [梯度 第 0..p/G 部分] + [优化器状态 第 0..p/G 部分]
GPU i: 完整参数 + [梯度 第 ip/G..(i+1)p/G 部分] + [优化器状态 对应部分]
```

**工作方式：** 反向传播时，梯度通过 Reduce-Scatter 而非 All-Reduce 聚合——每张 GPU 只收到自己负责的那部分梯度，用于更新自己负责的优化器状态和参数，再 All-Gather 同步完整参数。

**内存节省：** 梯度 + 优化器状态（约 $3/4$ 的内存）均切 $G$ 份，单卡节省更多。

### ZeRO Stage 3：分片优化器状态 + 梯度 + 参数

**连参数本身也切分**——每张 GPU 只持有 $1/G$ 的参数。

```
GPU 0: [参数 第 0..p/G 部分] + [梯度 第 0..p/G 部分] + [优化器状态 对应部分]
GPU i: [参数 第 ip/G..(i+1)p/G 部分] + [梯度对应部分] + [优化器状态对应部分]
```

**前向传播时：** 每层计算之前，用 All-Gather 广播该层完整参数；计算后立即丢弃非本卡负责的部分。

**可训练任意大小的模型**（只要总 GPU 数量够多），代价是每层前向/反向都有额外通信。

**三阶段对比：**

| 阶段 | 切分对象 | 单卡理论内存（$G$ 路数据并行）|
|------|---------|---|
| 基准数据并行 | 无切分 | $4P$（$P$ = 参数量，FP32）|
| Stage 1 | 优化器状态 | $4P - 2P(1-1/G) \approx 2P + 2P/G$ |
| Stage 2 | + 梯度 | $\approx P + 3P/G$ |
| Stage 3 | + 参数 | $\approx 4P/G$（近似线性减少）|

## 算法与系统协同设计

### 算法 / 硬件 / 分布式系统三者的耦合关系

早期机器学习把三者分离：统计模型 → 优化算法 → 工程实现。现代大规模分布式 ML 中，三者深度耦合：

```
算法选择决定通信模式（ADMM 每轮只传参数；GD 每步传梯度）
         ↓
通信模式决定系统瓶颈（参数量大 → All-Reduce 成瓶颈）
         ↓
系统瓶颈反向影响算法设计（为什么 SGD 用小批量？大批量 All-Reduce 太慢）
         ↓
硬件结构影响可行并行策略（NVLink 高带宽 → 张量并行可行；PCIe 低带宽 → 只能流水线并行）
         ↓
模型结构影响最优并行方案（Transformer 的矩阵乘特别适合张量并行）
```

**具体实例：** SGD 选用小批量，本质原因不只是统计收敛性，还因为小批量时通信量少（梯度可以和计算并行）；Ring All-Reduce 的设计从算法层面消除了参数服务器的单点通信瓶颈。

### 课程主线统一视角（从分布式 GD 到联邦学习的统一框架）

整个课程所有分布式算法都是同一个问题的不同解法：

$$\min_x \sum_{i=1}^N f_i(x) + g(x)$$

**用六个维度统一比较：**

| 算法 | 本地计算内容 | 通信对象 | 传输内容 | 允许不一致程度 | 状态归属 |
|------|---|---|---|---|---|
| 分布式 GD | 本地梯度（1 步）| 中心节点或邻居 | 梯度 $p$ 维 | 严格同步 | 中心节点维护参数 |
| Ring All-Reduce | 本地梯度 + Reduce | 环上两邻居 | 梯度段 $p/K$ 维 | 严格同步 | 各节点各有一份 |
| 一致性 ADMM | 解本地子问题（多步）| 主节点 | 参数 $p$ 维 | 每轮后一致 | 各节点有 $\theta_i$，主节点有 $z$ |
| 异步 SGD | 本地梯度（可能旧参数）| 参数服务器 | 梯度 $p$ 维 | 允许延迟 $\tau$ | 参数服务器 |
| FedAvg | 本地多步 SGD（$E$ epoch）| 服务器 | 模型参数 $p$ 维 | 多轮后才收敛 | 各客户端有本地模型 |
| ZeRO（Stage 3）| 本地层计算 + All-Gather | 数据并行邻居 | 参数段 $p/G$ 维 | 严格同步 | 切片分布在各 GPU |

### 分布式机器学习的发展趋势

**三条主线：**

1. **通信效率：** 从每步通信（GD）→ 多步后通信（ADMM/FedAvg）→ 压缩/稀疏化通信（ZeRO + 量化）→ 无通信（联邦学习本地训练）

2. **容错与异构：** 从严格同步（要求所有节点可靠）→ 异步（允许掉队者）→ 联邦学习（允许节点随时掉线）

3. **内存突破：** 从单卡（受显存限制）→ 数据并行（限于模型大小）→ 模型并行/流水线并行（打破单卡限制）→ ZeRO（消除冗余，线性扩展）

**算法-系统协同设计正成为主流：** Flash Attention（注意力机制重写，减少 HBM 访存）、混合精度训练（FP16 计算 + FP32 优化器状态）、Gradient Checkpointing（用重计算换内存），这些优化同时作用于算法和系统层面，无法归类为纯算法或纯工程问题。
