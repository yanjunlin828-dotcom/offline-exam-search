## 矩阵运算复杂度

### FLOP vs FLOPS（浮点运算次数与速率的区别）

- **FLOP（Floating Point Operation）**：一次浮点运算（如 `a + b` 是 1 FLOP，`a + b*c` 是 2 FLOPs），是与硬件无关的计量单位
- **FLOPS（FLOPs per Second）**：每秒能做多少次浮点运算，是硬件性能指标（如"100 TFLOPS"）

> 类比：FLOP 是"一块砖"，FLOPS 是"每秒能砌多少块砖"。注意大小写：`FLOPs`（小写 s）是 FLOP 的复数，`FLOPS`（大写 S）是速率，含义完全不同。

### 大 O 记号

描述"随数据规模 $n$ 增大，计算量如何增长"的语言。核心：**抓大放小，忽略常数和低阶项**。

| 复杂度 | 数据量扩大 10 倍后耗时变化 |
|--------|--------------------------|
| $O(n)$ | 变为 **10 倍** |
| $O(n^2)$ | 变为 **100 倍** |
| $O(n^3)$ | 变为 **1000 倍** — 矩阵运算常遇到，灾难性慢 |

### 向量点积 O(n)

$$u^Tv = \sum_{i=1}^n u_i v_i$$

$n$ 次乘法 + $n-1$ 次加法，总计 $O(n)$。数据量翻倍，工作量也翻倍。

### GEMV（矩阵向量乘）O(mn)

$A \in \mathbb{R}^{m \times n}$，$x \in \mathbb{R}^n$，计算 $y = Ax$（结果是 $m$ 维向量）。

结果有 $m$ 个元素，每个元素是一次长度为 $n$ 的点积 $O(n)$，共 $m$ 个，总计 $O(mn)$。

### GEMM（矩阵矩阵乘）O(mnk)

$A \in \mathbb{R}^{m \times k}$，$B \in \mathbb{R}^{k \times n}$，计算 $C = AB$（结果 $m \times n$）。

结果有 $mn$ 个元素，每个是长度 $k$ 的点积，总计 $O(mnk)$。**方阵特例**（$m=n=k$）：$O(n^3)$。

**复杂度对比表：**

| 运算 | 缩写 | 规模 | 复杂度 |
|------|------|------|--------|
| 向量点积 | — | $u,v \in \mathbb{R}^n$ | $O(n)$ |
| 矩阵向量乘 | GEMV | $A \in \mathbb{R}^{m \times n}$，$x \in \mathbb{R}^n$ | $O(mn)$ |
| 矩阵矩阵乘 | GEMM | $A \in \mathbb{R}^{m \times k}$，$B \in \mathbb{R}^{k \times n}$ | $O(mnk)$ |
| 方阵乘方阵 | GEMM 特例 | $A, B \in \mathbb{R}^{n \times n}$ | $O(n^3)$ |

**⚠️ 考试考点：多矩阵连乘先算哪个？**

计算 $A^TAx$（$A \in \mathbb{R}^{n \times p}$，$x \in \mathbb{R}^p$）：

- **方案一（错）：先算 $B = A^TA$，再算 $Bx$** → 复杂度 $O(np^2 + p^2) = O(np^2)$
- **方案二（对）：先算 $y = Ax$，再算 $A^Ty$** → 复杂度 $O(np + np) = O(np)$

**方案二快了 $p$ 倍！** 通用原则：**多矩阵连乘时，优先做维度小的（矩阵×向量），尽量避免矩阵×矩阵。**

## BLAS 库

### BLAS Level 1 / 2 / 3 分类（向量 / 矩阵向量 / 矩阵矩阵）

写 `C = A @ B` 时 Python 本身运行不快，但结果很快——因为 NumPy/SciPy 底层调用了 **BLAS（Basic Linear Algebra Subprograms）**，这是用 C/Fortran 写的高度优化的底层数学库（常见实现：OpenBLAS、Intel MKL）。

| 级别 | 运算类型 | 复杂度 |
|------|---------|--------|
| Level 1 | 向量-向量（点积、加法） | $O(n)$ |
| Level 2 | 矩阵-向量（GEMV） | $O(n^2)$ |
| **Level 3** | **矩阵-矩阵（GEMM）** | $O(n^3)$ |

Level 3 优化程度最深——GEMM 的计算量远大于数据读取量（计算密集型），非常适合 CPU 的向量化指令（SIMD）和缓存流水线优化。因此**不要自己写双重循环做矩阵乘法**，直接用 `@` 或 `np.dot`，底层会自动调用 BLAS。

### NumPy / SciPy 底层调用 BLAS

Python 是"翻译官"，BLAS 是"专业技工"。用户写 `A @ B`，NumPy 把这个请求翻译成 BLAS 的 DGEMM 函数调用，由 C/Fortran 代码完成计算。

## 内存布局

### 行优先（Row-Major）vs 列优先（Column-Major）

矩阵是二维的，内存是一维的，必须"拍平"存储：

- **行优先**（C/Python/NumPy 默认）：先存第一行，再存第二行，依此类推。矩阵 $\begin{bmatrix}1&2&3\\4&5&6\end{bmatrix}$ 在内存里是 `1,2,3,4,5,6`
- **列优先**（Fortran/MATLAB/R 默认）：先存第一列，再存第二列。同一矩阵在内存里是 `1,4,2,5,3,6`

### 遍历顺序与缓存命中率的关系

CPU 读取内存时，会把当前地址**附近的一段数据**一起装进高速缓存（Cache）。

- **缓存命中（Cache Hit）**：下一步要读的数据已在缓存 → 速度极快
- **缓存未命中（Cache Miss）**：不在缓存，要重新从慢速内存取 → 慢几十倍

如果遍历顺序与存储顺序**不一致**（如 NumPy 行优先但按列遍历），每次访问都缓存未命中，理论复杂度不变但实际时间慢几十倍。

### NumPy 默认行优先的陷阱

```python
# NumPy 默认行优先：内存里 1,2,3,4,5,6
A = np.array([[1,2,3],[4,5,6]])

# ❌ 按列遍历（访问顺序：1,4,2,5,3,6）——内存不连续，大量 Cache Miss
for j in range(3):
    for i in range(2):
        access A[i, j]

# ✅ 按行遍历（访问顺序：1,2,3,4,5,6）——内存连续，Cache Hit
for i in range(2):
    for j in range(3):
        access A[i, j]
```

**实际建议：用 NumPy 矩阵运算（`@`、`np.dot`），不要自己写遍历循环——NumPy 内部自动选择最优内存访问顺序。**

## 矩阵求逆 vs solve

### 为何禁止显式求 A⁻¹

OLS 公式 $\hat{\beta} = (X^TX)^{-1}X^Ty$ 看上去要先求逆，但这是**矩阵计算的两大忌讳之一**。

**原因一：数值不稳定。** 矩阵求逆会把条件数（衡量矩阵"病态"程度的指标）**平方放大**：

| 写法 | 相对误差量级 |
|------|------------|
| `solve(A, b)` | $\kappa \times \varepsilon_{machine}$ |
| `inv(A) @ b` | $\kappa^2 \times \varepsilon_{machine}$ |

当矩阵存在多重共线性（$\kappa$ 很大）时，求逆后的结果会变成"垃圾"。

**原因二：稀疏矩阵变稠密。** 稀疏矩阵（99% 是零）的逆通常是稠密矩阵（100% 非零），内存占用膨胀 100 倍，直接撑爆内存。

### np.linalg.solve vs np.linalg.inv

```python
# ❌ 错误：显式求逆
beta = np.linalg.inv(X.T @ X) @ (X.T @ y)

# ✅ 正确：转化为解线性方程组
A = X.T @ X    # p×p 对称正定矩阵
b = X.T @ y    # p 维向量
beta = np.linalg.solve(A, b)   # 内部用矩阵分解，不求逆
```

`np.linalg.solve(A, b)` 内部自动选合适的分解方式，数值稳定，效率高。

### 正规方程 (XᵀX)β = Xᵀy 的正确解法

**记忆口诀：写公式用 $A^{-1}b$，写代码用 `solve(A, b)`。**

数学上 $A^{-1}b$ 和 `solve(A, b)` 给出相同结果，但计算方法完全不同。`solve` 内部用矩阵分解（LU/Cholesky）加回代，既稳定又高效。

## 矩阵分解

矩阵分解的核心思路：把复杂矩阵 $A$ 分解成简单三角矩阵的乘积，再用**回代法（$O(n^2)$）**高效求解，把 $O(n^3)$ 的分解代价分摊到多次求解上。

### 三角方程组：后向/前向替换法

理解"为什么分解后求解快"的关键——三角方程组可以直接用逐行回代在 $O(n^2)$ 内解出。

**后向替换（Backward Substitution）——解上三角方程组 $Ux=y$：**

$$x_n = \frac{y_n}{u_{nn}}, \qquad x_k = \frac{y_k - \sum_{j=k+1}^{n} u_{kj}\,x_j}{u_{kk}} \quad (k = n-1, \ldots, 1)$$

从最后一行开始向上逐行代入，计算量为 $1 + 2 + \cdots + n$，复杂度 $O(n^2)$。

**前向替换（Forward Substitution）——解下三角方程组 $Ly=b$：** 从第一行向下逐行代入，同样 $O(n^2)$。

**工程意义：** 分解一次（$O(n^3)$）后，每次求解只需两次替换（$O(n^2)$）。这正是 ADMM 中 Cholesky 预分解能大幅加速的根本原因。

### Cholesky 分解：A = LLᵀ（SPD 矩阵）

**适用条件：** $A$ 是**对称正定矩阵（SPD）**。课程中最典型的就是 $X^TX$（只要 $X$ 列不完全线性相关，$X^TX$ 一定是 SPD 矩阵）。

**分解形式：** $A = LL^T$，其中 $L$ 是下三角矩阵，$L^T$ 是上三角矩阵。

**为什么比 LU 快：** LU 要维护 $L$ 和 $U$ 两个矩阵；Cholesky 利用对称性只算半个矩阵（$U = L^T$），复杂度仅 $\frac{1}{3}n^3$，是 LU 的**一半**。

**求解流程：** $LL^Tx = b$ → 先前向替换解 $Ly = b$，再后向替换解 $L^Tx = y$，各 $O(n^2)$。

### LU 分解：A = PLU（一般方阵）

**适用条件：** $A$ 是一般非奇异方阵（无特殊结构要求）。

**分解形式：** $A = PLU$，$P$ 是置换矩阵（数值稳定性选主元用），$L$ 下三角，$U$ 上三角。

**复杂度：** 分解 $\frac{2}{3}n^3$，之后每次求解只需 $O(n^2)$ 回代。

**求解流程：** $Ax = b \Rightarrow LUx = b$ → 令 $y = Ux$，先解 $Ly = b$（前向替换），再解 $Ux = y$（后向替换）。

### QR 分解（超定线性系统）

**适用条件：** $X \in \mathbb{R}^{n \times p}$，$n > p$（行数多于列数，超定系统）。

**分解形式：** $X = QR$，$Q$ 是正交矩阵（$Q^TQ = I$），$R$ 是上三角矩阵。实际用"经济型 QR"：$X = Q_1 R_1$（$Q_1 \in \mathbb{R}^{n \times p}$）。

**为什么数值最稳定：** $Q$ 是旋转变换，条件数为 1，不放大误差。直接对 $X$ 分解而非对 $X^TX$ 分解，避免了条件数平方的问题。

**OLS 的 QR 推导：** 将 $X = Q_1R_1$ 代入正规方程 $X^TX\hat\beta = X^Ty$：
$$R_1^TQ_1^TQ_1R_1\hat\beta = R_1^TQ_1^Ty \xRightarrow{Q_1^TQ_1=I} R_1\hat\beta = Q_1^Ty$$
只需解上三角方程组 $R_1\hat\beta = Q_1^Ty$（后向替换，$O(p^2)$），完全避免构造 $X^TX$。

**Python 写法：** `beta, _, _, _ = np.linalg.lstsq(X, y, rcond=None)`

### Cholesky 预分解技巧：cho_factor + cho_solve

**核心思想：** 如果要对**同一个矩阵 $A$ 做多次求解**（右端项 $b$ 不同），只做一次分解，后续每次只用 $O(n^2)$ 的回代。

在 ADMM 中，$x$-update 每轮都要解形如 $(A + \rho I)x = b$ 的方程，矩阵 $A + \rho I$ 在迭代中不变，因此只需在第一轮分解一次：

```python
from scipy.linalg import cho_factor, cho_solve

c, low = cho_factor(A)          # 第一轮：O(n³) 分解，只做一次
for k in range(max_iter):
    x = cho_solve((c, low), b)  # 每轮：O(n²) 回代，极快
```

### 三种分解适用场景与复杂度对比

| 分解 | 适用矩阵 | 分解复杂度 | 数值稳定性 | 典型用途 |
|------|---------|-----------|-----------|---------|
| LU | 一般非奇异方阵 | $\frac{2}{3}n^3$ | 一般 | 通用求解 |
| Cholesky | 对称正定（SPD） | $\frac{1}{3}n^3$（最快） | 好 | OLS 正规方程、ADMM |
| QR | 任意矩形 $n \geq p$ | $\approx 2np^2$ | **最好** | OLS 直接求解、统计推断 |

## 分块矩阵

### 分块矩阵乘法规则

把大矩阵"切块"后，块与块之间的乘法**遵从和普通矩阵乘法完全相同的规则**。将 $X$ 按行分成 $k$ 块：

$$X = \begin{bmatrix}X_1 \\ X_2 \\ \vdots \\ X_k\end{bmatrix}, \quad X^T = \begin{bmatrix}X_1^T & X_2^T & \cdots & X_k^T\end{bmatrix}$$

### XᵀX 的可加性：XᵀX = Σ XᵢᵀXᵢ

$$X^TX = \begin{bmatrix}X_1^T & X_2^T & \cdots & X_k^T\end{bmatrix}\begin{bmatrix}X_1 \\ X_2 \\ \vdots \\ X_k\end{bmatrix} = \sum_{i=1}^k X_i^TX_i$$

同理：$X^Ty = \sum_{i=1}^k X_i^Ty_i$

**意义：** 每个 $X_i^TX_i$ 只依赖第 $i$ 块数据，可以在第 $i$ 个节点上**独立并行计算**，结果再求和即可。这是整门课"分布式设计"的**数学基础**：找到可加结构 → Map-Reduce 并行。

| 场景 | 可加结构 | 应用 |
|------|---------|------|
| OLS（Lec 7） | $X^TX = \sum X_i^TX_i$ | 分布式 OLS |
| Logistic 梯度（Lec 9） | $\nabla\ell = \sum \nabla\ell_i$ | 分布式梯度下降 |
| ADMM（Lec 12-14） | $\sum_i f_i(\theta_i)$ | 一致性 ADMM |

### 分布式矩阵计算的分块策略

```
Map 阶段（k 个节点完全并行）：
  节点 i：拥有 Xᵢ → 计算 Mᵢ = XᵢᵀXᵢ（p×p），dᵢ = Xᵢᵀyᵢ（p维）

Reduce 阶段（主节点）：
  XTX = M₁ + M₂ + ... + Mₖ     ← 传输的是 p×p 的小矩阵，不是原始大数据！
  XTy = d₁ + d₂ + ... + dₖ

求解：
  beta = np.linalg.solve(XTX, XTy)
```

关键：节点间只传输 $p \times p$ 的小矩阵，原始大数据 $X$ 始终留在各自节点，**通信量与 $n$ 无关**。

## 矩阵计算原则

### 原则一：用 solve 替代显式求逆

见"矩阵求逆 vs solve"一节，此为最重要原则。

### 原则二：利用矩阵结构（对称性、稀疏性、正定性）

**对角矩阵示例：** 计算 $WA$（$W$ 是对角矩阵），直接 `W @ A` 是 $O(n^2p)$；利用结构只需用 $W$ 的对角元素逐行缩放 $A$，复杂度降为 $O(np)$，快了 $n$ 倍。

此原则在 Logistic 回归 Hessian $H = X^TWX$ 的计算中直接体现——不显式构造 $n\times n$ 的对角矩阵 $W$，只用向量乘法。

### 原则三：预分解后复用

同一矩阵要做多次求解时，分解只做一次（$O(n^3)$），每次求解只做回代（$O(n^2)$）。ADMM 中的 Cholesky 预分解是典型应用。

### 原则四：避免冗余中间矩阵

多矩阵连乘按从右到左、"先矩阵×向量"的顺序，避免生成不必要的中间大矩阵，节省内存和计算量。

### 原则五：合并多组向量运算为一次矩阵运算

若需对同一矩阵做 $k$ 组独立运算，将各向量拼成矩阵后**一次性调用**，效率显著高于循环 $k$ 次：

```python
# 计算 Ax₁, Ax₂, ..., Axₖ → 合并为矩阵乘
Y = A @ X                           # X = [x₁, x₂, ..., xₖ]，一次 GEMM

# 解 A⁻¹x₁, ..., A⁻¹xₖ（如 ADMM 多右端项）→ 合并为矩阵方程
X_sol = np.linalg.solve(A, X)       # 内部只做一次分解，k 次回代
```

原理：矩阵-矩阵乘（BLAS Level 3）的硬件利用率远高于多次矩阵-向量乘（Level 2），实际速度差异可达数倍。
