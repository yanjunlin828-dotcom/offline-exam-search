# CLAUDE.md — 离线资料检索引擎

## 这个项目是什么

一个**离线、本地**的资料检索工具，面向上机考试：对一个文件夹里的 `.md` 和 `.ipynb` 笔记/代码做高精度检索，能区分"概念定义"与"代码实现"，支持中文短概念，结果在浏览器里带章节路径和关键词高亮。

详细需求与设计在仓库内的 **`离线资料搜索引擎-工程方案v2_概念设计.md`**（下称"方案"）。**动手前请先完整读它**——它是规格的源文件，这份 CLAUDE.md 只提炼"最容易错、且错了就白干"的部分。方案是被一轮严格技术评审改过的第二版，里面记录了为什么这么选；如果你发现其中有更优解或仍有错误，**直接提出并说明理由**，不要默默照搬。

---

## 唯一的成败前提（底线，不是偏好）

运行环境是**考试机：不可控、无网络、临场不可修**。一旦"装不上 / 版本不对 / 没编进某特性"，整个工具在最需要的时候就是废的。由此推出三条不可违背的硬约束：

1. **只用 Python 标准库**。不引入任何第三方 pip 包作为运行依赖。
2. **不依赖任何编译期特性**。典型反例就是 SQLite 的 FTS5 / trigram —— `import sqlite3` 成功 ≠ 它可用，且版本不够无法临场升级。所以**检索引擎是纯 Python 倒排索引 + BM25**，不是 FTS5。
3. **前端 JS 本地打包**（`marked.js` / `highlight.js` 等放进仓库），绝不在运行时访问 CDN。

除这三条外，实现方式高度自由。

## 必须守住的正确性不变量

这些是会导致"静默 bug / 灵异查不到 / 引噪"的点，方案里有展开，这里只列要点供你随时核对：

1. **索引与查询必须共用同一个 Tokenizer**。切词规则一旦不一致，就会出现"明明有却查不到"。这是第一不变量。
2. **中文用字符 bigram 分词**（覆盖`范数``凸性`这类两字概念）。单字中文查询是**已知不支持项**，除非显式开启 unigram 选项——不要默默改变这个取舍。
3. **Markdown 解析：先判代码围栏状态，再判标题**。围栏（```` ``` ````/`~~~`）内必须**完全关闭**标题识别，否则代码里的 `# 注释` 会被误当标题。不假装完整兼容 CommonMark：保证围栏这一常见情形正确即可，缩进代码块等边角可作可选增强。
4. **`.ipynb` 当 JSON 解析**：`cell["source"]` 兼容 `str` 与 `list[str]`；markdown cell 也要跑标题解析；只索引 `source`、**忽略 `outputs`**；对老格式（无 `cells` 而有 `worksheets`）加 guard。
5. **别名展开：组内合取（AND）、组间析取（OR）**。一个别名（如全称短语）要求其全部 token 都在块内才算命中；不要把它拆成散词 OR，否则含单个常见词就误召。
6. **BM25 用 Lucene 式 IDF**（`ln(1 + (N - df + 0.5)/(df + 0.5))`），避免常见词出现负 IDF。
7. **片段高亮：先 HTML 转义，再包 `<mark>`**，顺序不能反。
8. **意图识别只做软偏好 + 可手动覆盖，不静默硬过滤**。查"定义"时优先 prose、并把前端类型开关切到对应档，但不要因为启发式误判就把结果整片藏掉。

## 验收标准 —— 请落地为自动化测试

建一个**小型 fixture 语料**（几篇 `.md` + 一两个 `.ipynb`），内容要刻意覆盖下面的刁钻情形，然后把这些金标准查询写成可重复运行的测试：

| 查询 | 期望 |
|---|---|
| `ADMM 定义` | ADMM 定义的 prose 块排第 1；prose 档下不混入实现代码 |
| `范数` / `凸性`（两字中文） | 能召回（验证 bigram） |
| `交替方向乘子法` 及全称 | 与 `ADMM` 同组；含**单独** `method` 的无关块**不**出现（验证合取展开） |
| `ADMM 实现` | 优先返回 code 块 |
| 带特殊字符的查询（如 `a AND "b*`） | **不报错**，按普通字符处理 |
| 语料中代码块含 `# 定义 ...` | 该行**不**被当成标题（验证围栏状态机） |
| 大小写 / 中英混写 | 不影响召回 |

性能目标（这个语料规模下应轻松达到）：构建数秒~数十秒，单次查询 < 1 秒。

## 建议的对外接口（可按需调整，非强制）

```
python build_index.py <materials_dir> -o index.pkl     # 构建：解析→分词→倒排→序列化
python search.py "<query>" [--type prose|code|all]      # P0：命令行检索
python server.py --index index.pkl --port 8000          # P1：本地 Web（stdlib http.server）
```

实现顺序也建议按方案的阶段走：**P0**（Tokenizer + Parser + 倒排 + BM25 + 命令行）能跑就先交付；**P1** 再加 Web 界面、意图识别、别名；P2 打磨。每个阶段都要可独立运行。

## 明确不在范围内（不要镀金）

- 语义/向量检索。
- 把 FTS5 当默认引擎（若你确有理由做成"检测到可用才启用"的可选加速，必须先能力自检、失败即降级，且不能改变纯 Python 路径的行为——但默认不做）。
- 完整 CommonMark 解析。
