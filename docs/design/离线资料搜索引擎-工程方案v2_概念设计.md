# 离线资料检索引擎 —— 工程方案 v2（修订版）

> 面向上机考试：**无网络、考试机不可控、临场不可修**的前提下，对一文件夹的 `.md` / `.ipynb` 资料做**高精度**检索，能区分"概念定义"与"代码实现"，支持中文短概念，结果可视化清晰。

---

## 0. 本版与 v1 的差异（以及为什么改）

v1 有一个真实失误：它声称"P0 全程标准库"就消除了"无网络/没装包"的风险。但 **FTS5 和 trigram 分词器不是 pip 包，而是 SQLite 的编译期特性**——`import sqlite3` 成功，不代表底层 SQLite 编进了 FTS5，更不代表版本够新（trigram 需 SQLite **3.34.0+**，2020-12 才发布；部分 Linux 仍用 3.26）。在 Python 上换新版 SQLite 通常要重新编译 Python，考试机上做不到。也就是说 v1 把最关键的 P0 路径，押在了唯一一个"编译期不确定、临场修不了"的环节上。

v2 的核心调整：

1. **引擎换成纯 Python 倒排索引 + BM25，作为唯一基线**。零编译期依赖、只用标准库、保证能跑。
2. **中文用字符 bigram 分词**，绕开 trigram 的"查询词需 ≥3 字符"限制，让"范数""凸性"这类两字概念也能命中。
3. 纯 Python 引擎自己掌控查询解析，**"用户输入注入 MATCH 语法"的问题直接消失**（不再有 FTS5 查询语法可被注入）。
4. Markdown 解析改为**围栏状态机**，根治"代码块里的 `# 注释` 被当成标题"。
5. 别名展开改为**组内合取（AND）、组间析取（OR）**，解决"展开成散词导致引噪"。
6. **意图关键词识别从 P2 提前到 P1**，让"查定义优先给散文"在有可视化时就生效。
7. **FTS5 降级为可选加速项**，且明确写清它的版本/编译/中文三重限制——不建议为它引入第二条代码路径。

## 1. 目标与非目标

**要做到：**
- 输入 `ADMM`，精确定位其**定义**段落，而非把所有提到它的代码也堆上来。
- 中文短概念（如`范数`、`凸性`）能召回。
- 同一概念多种写法（`ADMM` / `交替方向乘子法` / 全称）都能召回，且不引噪。
- 按意图区分：查"定义"优先给散文，查"实现"优先给代码。
- 结果带章节路径面包屑，关键词高亮。
- 全程离线、低配可跑、**保证启动**。

**不做（控制错误面）：**
- 不做语义向量检索（低配 + 求稳，结构化已够）。
- **P0/P1 不引入 FTS5 双后端**（双引擎会带来行为不一致与额外 bug，与"不引入新错误"冲突）。
- 不追求完整 CommonMark 兼容（只保证围栏代码块这一最常见情形正确，见 §5）。

## 3. 架构总览

```
构建期（offline，一次性，可重复运行）
  资料文件夹
      │  Parser（md 围栏状态机 / ipynb JSON）
      ▼
  结构化块 chunk（标题、正文、类型、所属路径、段内序号）
      │  Tokenizer（英文词 + 中文 bigram）→ Indexer
      ▼
  内存倒排索引 → 序列化到磁盘 index.pkl
      ＋ aliases.txt（人工维护别名）

查询期（每次搜索，纯内存）
  查询串
      │  同一个 Tokenizer 切词 + 意图识别 + 别名展开
      ▼
  倒排检索候选块 → BM25 打分 → 叠加加分（标题/类型/位置）→ 排序
      ▼
  结果（章节面包屑 + 高亮片段）──► 浏览器（stdlib http.server）
```

**关键点：构建期和查询期共用同一个 Tokenizer**——查询和索引的切词规则必须完全一致，否则会出现"明明有却查不到"的灵异问题。这是整套设计第一条不可违背的约束。

## 4. 分词器（Tokenizer）

统一规则，索引与查询都走它：

1. 先转小写（仅影响英文/数字）。
2. 扫描字符：
   - 连续的 `[a-z0-9]+` → 一个**英文词 token**。
   - 连续的中日韩表意文字（基本区 `U+4E00–U+9FFF`，可选含扩展区 `U+3400–U+4DBF`）→ 输出**重叠 bigram**（如`范数学`→`范数`,`数学`）；该连续段长度为 1 时输出该单字。
   - 其余字符（标点、空白）→ 分隔符，丢弃。

**为什么用 bigram 而不是 trigram**：trigram 要求查询词 ≥3 字符，`范数``凸性`这种两字概念会直接召不回；bigram 把"≥2 字"都覆盖了，正好匹配中文概念词的常见长度。

**已知取舍（如实说明）**：bigram 索引下，**单个汉字的查询**（1 字）无法命中按 bigram 建索的文本——这是有意的取舍，单字检索语义本就模糊。若确实需要，可在分词器里**额外**对每个汉字补一个 unigram token（代价是索引略大），作为一个清晰隔离的可选开关，不进基线。

> 副作用：bigram 是字符级近似匹配，偶尔会因为"两个 bigram 都恰好出现"产生轻微过召回；这由 BM25 排序 + 别名合取（§6）兜住，召回优先、精度靠排序，符合目标。

## 5. 解析器（Parser）

### 5.1 Markdown —— 围栏状态机（根治标题误判）

逐行扫描，维护一个布尔状态 `in_fence`：

- **进入围栏**：当 `in_fence=False` 且某行（去除前导空格后）以 `` ``` `` 或 `~~~` 开头（其后可跟语言标注如 `python`）。记住围栏字符与长度。
- **退出围栏**：当 `in_fence=True` 且某行是**同种字符、长度不小于开围栏**的纯围栏行。
- **`in_fence=True` 期间，完全关闭标题识别**——这样代码块里的 `# 定义 ADMM` 不会被误判成标题。
- 仅当 `in_fence=False` 时，`^#{1,6}\s` 才算 ATX 标题，用于维护标题栈、生成 `heading_path`。

围栏内的内容标 `code`，围栏外的散文标 `prose`。

**坦白边界**：4 空格缩进代码块、列表内代码等 CommonMark 细则不在零依赖基线的保证范围内（手写正则很难完全正确，硬做反而易引新错）。如目标机恰好装了 `markdown-it-py` 或 `mistune`，可作为**可选**的更严谨解析后端；否则围栏状态机已覆盖笔记里 99% 的情形。**不假装手写解析器完全符合 CommonMark。**

### 5.2 Notebook（.ipynb）—— 当 JSON 解析

- 用标准库 `json` 读取，遍历 `cells`。
- `cell["source"]` **可能是字符串，也可能是字符串列表**，统一 `"".join(...)` 兼容。
- `markdown` cell：**跑与 `.md` 完全相同的标题解析**，按标题切块、生成 `heading_path`，不要整个 cell 当一个 prose 块。
- `code` cell：标 `code`，**只索引 `source`，忽略 `outputs`**（执行输出含大段 base64 图片/报错，纯噪音）。
- 老格式 guard：若顶层没有 `cells` 而有 `worksheets`（nbformat v3），从 `worksheets[].cells` 取（边缘情形，加个判断即可）。

## 6. 数据模型与索引

**chunk（检索单位）字段：**

```
chunk {
  id            : int                  # 自增
  file          : str                  # 来源相对路径
  heading       : str                  # 本块所属标题文本
  heading_path  : str                  # 如 "优化方法 > ADMM > 定义"
  text          : str                  # 原始正文（用于片段展示与高亮）
  body_tokens   : list[str]            # text 经 Tokenizer 切出的 token（含标题 token）
  heading_tokens: set[str]             # heading 经 Tokenizer 切出的 token（用于标题加分）
  ctype         : "prose" | "code"
  position      : int                  # 本块在所属章节内的序号（0=紧跟标题）
}
```

**倒排索引（内存，纯 Python dict）：**

```
inverted : dict[token -> list[(chunk_id, tf)]]   # tf = 该 token 在该块出现次数
df       : dict[token -> int]                    # 含该 token 的块数
chunks   : list[chunk]
N        : int                                   # 块总数
avgdl    : float                                 # 平均块长度（token 数）
```

构建完成后用标准库 `pickle` 序列化到 `index.pkl`（自建本地数据，可信，pickle 最省事；若顾虑可改 `json`，注意 json 会把 int 键转成 str）。查询期启动时一次性 `load` 进内存。

**规模与性能（这个量级下完全够）**：你的语料是自己的几 MB~几十 MB、几十~几百文件。建索引（读盘+分词+建表）约数秒到数十秒，一次性；查询是几次 dict 查找加打分，毫秒级。内存占用几十到一两百 MB，低配可承受。

## 7. 排序

### 7.1 BM25（核心相关度）

对查询 token 集合 `Q` 与块 `D`，用标准 Okapi BM25（采用 Lucene 形式的 IDF，避免常见词出现负 IDF）：

```
IDF(t)      = ln( 1 + (N - df(t) + 0.5) / (df(t) + 0.5) )

score(D,Q)  = Σ_{t in Q}  IDF(t) * ( tf(t,D) * (k1 + 1) )
                          ----------------------------------------------
                          ( tf(t,D) + k1 * (1 - b + b * |D| / avgdl) )
```

- `|D|` = 该块 token 数；`avgdl` = 全部块的平均 token 数。
- 参数取安全默认：`k1 = 1.5`，`b = 0.75`。
- 标题 token 已并入块的 token 袋，所以标题命中本身就会贡献 BM25。

### 7.2 叠加加分（在 BM25 之上做加性 re-rank，每项都简单可验证）

```
final = score_bm25
      + W_head  * (查询主题 token 命中 heading_tokens 的个数 / 主题 token 数)
      + W_type  * [ctype 与意图一致 ? 1 : 0]
      + W_pos   * 1 / (1 + position)          # 越靠近标题（position 越小）越像定义
```

建议初值 `W_head=2.0, W_type=1.5, W_pos=0.5`，用 §11 的金标准查询微调。

> 说明：`W_pos` 是弱启发式（有些定义前有引子），权重克制，不让它扛大梁。

## 8. 查询流程（端到端）

1. **切词**：查询串过同一个 Tokenizer → 得到查询 token。
2. **意图识别（P1 起生效）**：查询若含意图词，设定类型偏好：
   - `定义 / 概念 / 是什么 / 含义 / 介绍` → 偏好 `prose`
   - `实现 / 代码 / 怎么写 / 例子 / 用法` → 偏好 `code`
   - 该偏好**默认作为加分项 + 自动把前端的类型开关切到对应档**，但**不静默硬过滤**（避免启发式误判时把结果整片藏掉）；用户可在前端手动覆盖。
3. **别名展开（精确版）**：主题词查 `aliases.txt`。每个别名是一个**短语**，命中规则为：
   - **组内合取**：一个别名的所有 token 必须都出现在该块（如全称 `alternating direction method of multipliers` 的四个词全在，才算命中这个别名）——这样含单个 `method` 的块不会被误召。
   - **组间析取**：同一概念的各种写法之间取 OR。
   - （可选精度升级：要求别名 token 相邻，需要在倒排里存 token 位置；非必须。）
4. **检索候选**：取查询/别名 token 的倒排表并集，得到候选块集合。
5. **打分**：对候选块算 §7 的 `final`。
6. **过滤/排序**：按前端类型开关（全部 / 只看概念 / 只看代码）过滤，按 `final` 降序排。
7. **返回**：每条含 `heading_path`、`file`、片段、`ctype`。

> 因为查询解析完全在 Python 内做，用户随手输入的 `"`、`*`、`AND` 只会被当普通字符切词，**不存在 FTS5 那种查询语法注入/报错问题**——这是换引擎顺带消掉的一个 P0 隐患。

## 9. 前端 / 可视化

- **服务**：标准库 `http.server`（`BaseHTTPRequestHandler`）。索引在服务启动时 load 进全局，handler 提供两个路由：`/` 返回静态页，`/search?q=...&type=...` 返回 JSON。单用户够用；要并发可换 `ThreadingHTTPServer`（标准库，3.7+）。零 pip。
- **渲染**：`marked.js` 渲染 markdown，`highlight.js` 做代码着色，**两者考前下载、本地打包**进 `web/`，不走 CDN。
- **结果卡片**：顶部**面包屑标题路径**（`文件 > 章 > 节`），下面是高亮片段；按文件分组折叠。
- **高亮**：在片段里按**用户原始查询子串**做字面匹配高亮——**先 HTML 转义片段文本，再包 `<mark>`**，顺序别反（先转义后包标签），否则会破坏渲染或引入注入。
- **类型开关**：全部 / 只看概念 / 只看代码，与 §8 意图识别联动。

| 用例 | 期望 |
|---|---|
| 查 `ADMM 定义` | ADMM 定义的 prose 块排第 1；prose 档下实现代码不出现 |

## 13. 关于 FTS5 的明确说明（为什么只当可选项）

如果将来想用 FTS5 加速，必须知道并接受以下三点，否则会把 v1 的坑又踩回来：

1. **可用性不保证**：FTS5 是 SQLite 编译期特性，`import sqlite3` 成功 ≠ FTS5 可用。**用前必须做能力自检**：`try` 建一张 `fts5(...)` 临时表，失败立刻降级到纯 Python，绝不让它在查询时才崩。
2. **版本门槛**：trigram 分词器需 SQLite **3.34.0+**；老系统（如 3.26）不可用，而在 Python 上升级 SQLite 通常要重编 Python，考试机做不到。
3. **中文限制**：trigram 要求查询词 ≥3 字符，`范数``凸性`等两字中文召不回；非 trigram 的内置分词器（unicode61/ascii）对无空格中文基本无效。**就中文而言，纯 Python bigram 反而比 FTS5 更合适。**

结论：**FTS5 在这个场景里收益有限、风险集中，且会引入第二条与主引擎行为不一致的代码路径。除非语料极大且确认环境，否则不建议加。**

> 另：FTS5 的 `bm25(table, w0, w1, ...)` 权重是按表的列顺序依次映射的（含 UNINDEXED 列）。本方案不依赖它，此处仅备查。

## 15. 给实现者的"易错点清单"（写代码时逐条核对）

1. 查询与索引**必须用同一个 Tokenizer**，否则查得到查不到全凭运气。
2. Markdown 解析**先判围栏、再判标题**；围栏内禁用标题识别。
3. `ipynb` 的 `source` 兼容 str 与 list；只取 `source`、忽略 `outputs`；markdown cell 也要解析标题。
4. 别名展开**组内 AND、组间 OR**，别 OR 一堆散词。
5. BM25 用 Lucene 式 IDF（`ln(1 + ...)`），避免负 IDF。
6. 中文 bigram 解决两字词；单字查询是已知不支持项（除非开 unigram）。
7. 高亮**先 HTML 转义、再包 `<mark>`**。
8. 若用 FTS5：**先能力自检再用，失败即降级**；用户输入进 MATCH 必须转义（纯 Python 引擎则无此问题）。
9. 交付前**在目标机实测构建+查询全流程**。
